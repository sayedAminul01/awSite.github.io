Complete VPS Security Guide: Fortify Your Server Step-by-Step

Learn how to properly secure your VPS before deploying applications. A comprehensive guide for beginners to advanced users.

By: Alex Chen Updated: March 15, 2023 Reading Time: 15 min

As a founder and sysadmin mentor, I've seen too many servers compromised due to basic security oversights. In this guide, I'll walk you through the essential steps to transform your vulnerable VPS into a fortress before you deploy your application.

1

Initial Connection to Your VPS

First, connect to your VPS using SSH from your local machine:

Connect via SSH
ssh root@YOUR_SERVER_IP

Why This Matters:

This is your first interaction with the server. Replace YOUR_SERVER_IP with your actual server IP address. If this is your first time connecting, you'll be asked to accept the server's fingerprint - this is a security measure to ensure you're connecting to the right server.

Pro Tip:

If you're using a cloud provider like DigitalOcean, AWS, or Vultr, they usually provide the IP address and initial root password in your control panel.

2

Change the Root Password

Even if your VPS provider gave you a secure password, change it immediately:

Change Root Password
passwd

Why This Matters:

Your VPS provider might have generated a password that could be predictable or reused. Creating your own strong, unique password ensures that only you have access to the root account.

Password Best Practices:

  • Use at least 12 characters
  • Mix uppercase, lowercase, numbers, and symbols
  • Avoid dictionary words and personal information
  • Consider using a password manager
3

Create a New User Account

Using the root account for daily tasks is dangerous. Create a regular user with sudo privileges:

Create New User
adduser alex
Grant Administrative Privileges
usermod -aG sudo alex

Why This Matters:

The root account has unlimited power over your system. A single typo as root can destroy your entire system. By creating a regular user with sudo privileges, you add a layer of protection - you'll need to prefix destructive commands with sudo, making you think twice before executing them.

4

Disable Root SSH Login

Prevent direct root access via SSH to protect against brute-force attacks:

Edit SSH Configuration
sudo nano /etc/ssh/sshd_config

Find and modify these lines:

SSH Configuration Changes
PermitRootLogin no
PasswordAuthentication no

Why This Matters:

Root is the most targeted account by attackers. Disabling direct root SSH login forces attackers to guess both a username and password. Disabling password authentication entirely and requiring SSH keys makes your server virtually immune to brute-force attacks.

Important:

Make sure you've set up SSH key authentication for your new user (next step) before restarting SSH, or you'll lock yourself out of the server!

Restart SSH Service
sudo systemctl restart ssh
5

Set Up SSH Key Authentication

On your local machine, generate an SSH key pair:

Generate SSH Key
ssh-keygen -t ed25519 -C "alex@cyberfort"

Copy the public key to your server:

Copy Public Key to Server
ssh-copy-id alex@YOUR_SERVER_IP

Test the connection:

Test SSH Connection
ssh alex@YOUR_SERVER_IP

Why This Matters:

SSH keys are exponentially more secure than passwords. While a password can be guessed through brute force, SSH keys use cryptographic algorithms that are practically impossible to crack. The private key stays on your local machine, and only the public key is stored on the server.

Pro Tip:

If you want maximum security, protect your SSH key with a passphrase. For automated processes, you can use keys without passphrases, but store them securely.

6

Configure the Firewall (UFW)

Set up a firewall to control incoming and outgoing traffic:

Configure Firewall Rules
sudo ufw allow OpenSSH
sudo ufw allow 80
sudo ufw allow 443
sudo ufw enable

Check the firewall status:

Check Firewall Status
sudo ufw status

Why This Matters:

A firewall acts as a gatekeeper for your server, blocking all unnecessary ports. By default, UFW denies all incoming connections and allows all outgoing. We're specifically allowing SSH (port 22), HTTP (port 80), and HTTPS (port 443) while blocking everything else.

Port Explanation:

  • Port 22 (SSH): For secure remote administration
  • Port 80 (HTTP): For web traffic (will redirect to HTTPS)
  • Port 443 (HTTPS): For encrypted web traffic
7

Install Fail2Ban for Brute-Force Protection

Fail2Ban monitors log files and bans IPs that show malicious signs:

Install and Enable Fail2Ban
sudo apt install fail2ban -y
sudo systemctl enable fail2ban
sudo systemctl start fail2ban

Check if Fail2Ban is running:

Check Fail2Ban Status
sudo fail2ban-client status

Why This Matters:

Even with SSH key authentication, your server will still receive thousands of login attempts from bots scanning the internet. Fail2Ban automatically detects these repeated failed attempts and temporarily bans the offending IP addresses, reducing noise in your logs and protecting against dictionary attacks.

8

Keep Your System Updated

Regular updates patch security vulnerabilities:

Update System Packages
sudo apt update && sudo apt upgrade -y
sudo apt autoremove -y

Set up automatic security updates:

Enable Automatic Updates
sudo apt install unattended-upgrades -y
sudo dpkg-reconfigure unattended-upgrades

Why This Matters:

Software vulnerabilities are discovered daily. Unpatched systems are the #1 cause of security breaches. Regular updates ensure you have the latest security patches. Automatic updates for security patches ensure you're protected even if you forget to manually update.

9

Disable Unused Services

Reduce your attack surface by disabling unnecessary services:

List Enabled Services
sudo systemctl list-unit-files --type=service --state=enabled

Disable any unnecessary services:

Disable Unnecessary Service
sudo systemctl disable SERVICE_NAME

Why This Matters:

Every running service is a potential entry point for attackers. By disabling services you don't need, you reduce your "attack surface" - the number of ways someone can potentially compromise your system. Common services to consider disabling include FTP, telnet, and unnecessary database services if you're not using them.

10

Monitor Login Attempts

Regularly check your authentication logs for suspicious activity:

Check Failed Login Attempts
sudo cat /var/log/auth.log | grep "Failed"

Monitor logs in real-time:

Real-time Log Monitoring
sudo tail -f /var/log/auth.log

Why This Matters:

Monitoring your logs helps you understand the threats your server faces. You'll be able to see attempted attacks, identify patterns, and take additional protective measures if necessary. Regular monitoring is a key part of security hygiene.

+

Bonus Security Measures

For enhanced security, consider these additional measures:

Install Rootkit Scanner
sudo apt install rkhunter -y
sudo rkhunter --update
sudo rkhunter --check
Restrict SSH to Specific IP (If Static)
sudo ufw allow from YOUR_IP to any port 22

Why These Matter:

RKHunter scans for rootkits and other malware that might have bypassed your other security measures. Restricting SSH to your specific IP address (if you have a static IP) completely blocks SSH access from any other location, dramatically reducing your attack surface.

Advanced Security Considerations:

  • Set up intrusion detection with AIDE or similar tools
  • Configure automatic security audits
  • Implement two-factor authentication for SSH
  • Set up centralized logging
  • Regularly backup your configuration and data

Conclusion

By following these steps, you've transformed your vulnerable VPS into a secure fortress. Your server now has:

  • Protected administrative access
  • Firewall protection
  • Brute-force attack prevention
  • Automatic security updates
  • Reduced attack surface

Remember that security is an ongoing process, not a one-time setup. Regularly review your security measures, monitor logs, and stay informed about new threats. Your VPS is now ready for application deployment with a solid security foundation.
